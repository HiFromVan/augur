# 技术选型

## 一、总体原则

- **小团队友好**：选主流、文档好、招人容易的技术
- **迭代速度优先**：MVP 快速验证，避免过早优化
- **数据管道可靠**：数据是核心资产，存储和更新不能出错
- **AI 部分务实**：不追求最新模型，追求可解释、可调试

## 三、各层详细选型

### 3.1 后端 API — FastAPI

```python
# 示例接口设计
GET /api/matches?date=2026-04-05&league=csl
→ 返回当日中超比赛列表 + 预测概率

GET /api/matches/{match_id}
→ 返回单场详情：预测、近期状态、历史交锋

GET /api/teams/{team_id}/ratings
→ 返回球队当前 Pi-Ratings 历史曲线
```

**选 FastAPI 的理由：**
- Python 生态，和 ML 代码无缝整合
- 自动生成 Swagger 文档
- async 支持，爬虫/IO 密集任务友好
- 比 Django 轻，比 Flask 规范

### 3.2 前端 — Next.js

**选型理由：**
- SSR/SSG 支持，比赛数据可以静态化，减少服务器压力
- App Router 架构适合数据展示型页面
- 与 Tailwind CSS 配合，移动端适配快

**UI 组件库：** shadcn/ui（可定制度高，无额外依赖）

**数据可视化：** Recharts（轻量，React 生态，满足概率条/折线图需求）

### 3.3 ML 层

| 工具 | 用途 |
|------|------|
| `catboost` | 主预测模型 |
| `scikit-learn` | 特征预处理、校准 (`CalibratedClassifierCV`) |
| `pandas` | 特征工程、数据处理 |
| `numpy` | 数值计算 |
| `optuna` | 超参数调优 |
| `mlflow` | 实验追踪、模型版本管理 |
| `joblib` | 模型序列化 |

**模型服务方式：** 直接在 FastAPI 进程中 `joblib.load` 模型文件，推理时间 <10ms，无需独立 model server（数据量不大的情况下）。

### 3.4 数据采集

```python
# 技术选型
httpx + BeautifulSoup  # 轻量爬虫，处理单页面
Scrapy                 # 复杂多页面爬虫，有中间件/去重支持
APScheduler            # 定时任务（不用 Celery，项目早期过重）
```

**反爬应对：**
- 随机 User-Agent 池
- 请求间隔 2-5 秒（礼貌爬虫）
- 代理 IP（如遇封禁，V2 阶段处理）

### 3.5 数据库

**PostgreSQL** — 主数据库
- 比赛历史、球队数据、预测结果
- 支持 JSONB（灵活存储原始爬虫数据）
- 时序查询性能足够（数据量级：百万场比赛）

**Redis** — 缓存层
- 今日赛程缓存（TTL 1小时）
- 当前 Pi-Ratings 缓存（比赛后失效）
- 防止频繁重复计算

### 3.6 部署

**MVP 阶段（单机）：**
```yaml
# docker-compose.yml 结构
services:
  api:        # FastAPI
  worker:     # 定时任务 + 爬虫
  db:         # PostgreSQL
  redis:      # Redis
  nginx:      # 反向代理
```

**推荐服务器：** 阿里云 / 腾讯云 2核4G（¥100/月级别），足够跑 MVP。

**域名和备案：** 做 Web 版需要域名备案（约2-4周），小程序需要公司主体。

## 四、(V2) AI 问答技术选型

如果加入自然语言问答功能：

```
用户问题
    ↓
Claude API (claude-sonnet-4-6)
    ↓ （Function Calling）
查询数据库（比赛数据、球队状态、预测结果）
    ↓
组合上下文 → 生成回答
```

**关键设计：** AI 不直接生成概率，概率来自模型计算，AI 负责解释和整合文字描述。

## 五、开发顺序建议

```
Week 1-2: 数据管道
  → 爬虫采集中超历史数据（至少3个赛季）
  → 数据库建表，数据清洗入库

Week 3-4: 特征工程 + 模型
  → 实现 Pi-Ratings 计算
  → 实现近期状态特征
  → CatBoost baseline，回测验证

Week 5-6: 后端 API
  → FastAPI 接口
  → 定时任务：每日更新数据 + 重新预测

Week 7-8: 前端
  → Next.js 比赛列表页
  → 比赛详情页
  → 移动端适配

Week 9+: 迭代
  → 赔率数据接入
  → 微信小程序
  → AI 问答
```

## 六、关键风险点

| 风险 | 描述 | 应对 |
|------|------|------|
| 数据源不稳定 | 国内网站可能改版或封 IP | 多数据源备份，存原始数据 |
| 中超数据稀缺 | 历史数据不如欧洲联赛丰富 | 用欧洲数据预训练 + 迁移学习 |
| 模型过拟合 | 历史数据有限，容易过拟合 | 严格时间切分，交叉验证 |
| 合规风险 | 彩票相关灰色地带 | 定位数据工具，不涉及资金流 |
